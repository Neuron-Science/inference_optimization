Overview of Multi-LoRA Serving
What is LoRA?
Low-Rank Adaptation (LoRA) is a parameter-efficient fine-tuning (PEFT) technique used to adapt large language models (LLMs) for specific tasks at a low cost. Compared to full parameter fine-tuning, which retrains all of the LLM parameters, LoRA freezes the original model weights and injects a small number of new, trainable parameters (adapters) into the model's layers. These adapters consist of two small matrices that are updated instead of the full weight matrix.

This method provides several key advantages, including computational and memory efficiency, as the size of LoRA adapters is typically less than 1% of the base model size. 
Empty Image
Multi-LoRA Serving
Base models can produce different LoRA adapters for different tasks. These adapters share the same base model weights and therefore are loaded into device memory (HBM) along with the base model weights for serving. However, HBM capacity is limited and often cannot host all of the LoRA adapters. Therefore, some adapters must be offloaded to CPU memory and dynamically loaded into HBM at runtime according to user requests. Multi-LoRA serving shares a single base model across many adapters, allowing one device to serve what would otherwise require dozens of separate model deployments.

Framework Support for Multi-LoRA Serving
Multi-LoRA serving has gained significant popularity as companies and researchers seek cost-effective ways to deploy specialized LLMs at scale. Rather than hosting separate full models for different tasks or users, multi-LoRA serving enables a single base model to be shared across multiple LoRA adapters. This approach has become particularly attractive because it dramatically reduces memory footprint and infrastructure costs. While a full model might require gigabytes of device memory, LoRA adapters typically occupy only megabytes, allowing tens or even hundreds of specialized models to coexist on the same hardware. The rise of multi-LoRA serving has been driven by increasing demand for personalized AI experiences, where different customers or use cases require domain-specific fine-tuning. Companies and researchers have developed sophisticated serving systems that can dynamically load and swap LoRA adapters with minimal latency, batching requests across different adapters to maximize throughput.

Open source frameworks have been instrumental in democratizing multi-LoRA serving and making it accessible to a broader audience. vLLM added native support for serving multiple LoRA adapters simultaneously, enabling dynamic LoRA adapter swapping in and out of GPU memory with minimal overhead. SGLang has emerged as a compelling alternative with its focus on efficient structured generation and co-design of frontend language with backend runtime, offering competitive multi-LoRA serving capabilities with optimizations for both throughput and flexibility in handling complex prompting scenarios. LoRAX (LoRA eXchange) was specifically designed for multi-LoRA serving with features like adapter preloading and optimized batching strategies. HuggingFace's Text Generation Inference (TGI) integrated multi-LoRA support to complement its production-grade serving capabilities. Other frameworks, such as Ray Serve, have enabled flexible multi-LoRA deployments at scale. These open source tools have introduced innovations such as continuous batching across adapters, efficient adapter caching strategies, and optimized CUDA kernels for LoRA computation, significantly reducing the barrier to entry for organizations wanting to deploy specialized models.

HuggingFace has also launched a popular open-sourced project ﻿PEFT﻿ for LoRA fine-tuning. Almost all popular LLMs in HuggingFace have LoRA adapters, including Llama3.1 8B Instruct (1060 adapters), QWen2.5 7B (634 adapters), and DeepSeek-R1 (122 adapters).

Neuron Support for Multi-LoRA
The Neuron science team predicted the popularity of multi-LoRA serving on GPUs before it drew significant attention from academia and industry. As a result, we began development to support multi-LoRA on Neuron in February 2024. When we looked into serving, we realized there was no support for fine-tuning and developed this feature first. LoRA fine-tuning is the first general module in NxD and has been used by internal applications, such as Flux fine-tuning on Neuron and QLoRA. 

Now (in 2025), we have an external customer, Databricks, requesting performant dynamic multi-LoRA serving as a feature. Thanks to our early efforts in supporting LoRA on Neuron, we have developed a pipeline of LoRA features and multi-LoRA serving is a representative application to showcase the advantages of our performant and cost-efficient AI accelerators.

Static Multi-LoRA Serving (Released on 04/03/2025)
After delivering LoRA fine-tuning in NxD, we began working on multi-LoRA serving in NxDI. We delivered the first milestone of static multi-LoRA serving support in November 2024 and integrated this feature into vLLM. With this feature, users can pre-allocate a fixed number of adapters in device memory before inference begins and use these adapters throughout serving. The adapters remain allocated throughout inference, but no new adapters can be used. This is simpler to implement with predictable performance, but typically only dozens of adapters can be supported based on their size and the available device memory. 
LoRA Module Layout
A model typically consists of multiple layers, where each layer has multiple modules, (e.g. query, key, value, output, up projection, gate projection, and down projection in the Llama model architecture). LoRA adapters can be applied to multiple modules across all of the layers in the model. For convenience, we denote a LoRA module as the set of LoRA ﻿AAA﻿ and ﻿BBB﻿ weights applied to the same base module and focus on one LoRA module for illustration.

There are many different parallelism strategies that can be applied to LoRA. We focus on two that Neuron supports. With data parallelism (DP), both the base model and LoRA adapters are replicated across all devices and each device processes different data batches. This requires each device to store a full copy of active LoRA adapters, leading to memory redundancy, but simplifying the implementation. With tensor parallelism (TP), individual weight matrices are split across devices within a layer. The base model weights are partitioned row-wise or column-wise and the LoRA matrices are partitioned accordingly. This reduces the per-device LoRA adapter memory footprint, but increases communication overheads

Empty Image
Empty Image


When the base module is a linear layer, both LoRA ﻿AAA﻿ and LoRA ﻿BBB﻿ are linear layers, i.e. both matrices are replicated across all devices. 




When the base module is a column-parallel layer, LoRA ﻿AAA﻿ is replicated and LoRA ﻿BBB﻿ is partitioned along the column dimension such that each column is processed by a different device. In Neuronx Distributed (NxD), the base module and LoRA ﻿BBB﻿ are of type ColumnParallelLinear.



When the base module is a row-parallel layer, LoRA ﻿AAA﻿ is partitioned along the row dimension such that each row is processed by a different device and ﻿BBB﻿ is replicated. In Neuronx Distributed (NxD), the base module and LoRA ﻿AAA﻿ are of type RowParallelLinear.
In general, even when tensor parallelism is applied to the base module, one of the LoRA matrices must be a linear layer even if tensor parallelism is applied to the other.
Memory Management and Workflow
Neuron provides support for both static and dynamic multi-LoRA serving. In both settings, compared to LLM serving with the base model only, multi-LoRA serving incurs an additional device memory (HBM) footprint in order to host LoRA adapters as well as extra overhead for LoRA computation.
Empty Image

Since Neuron expects static tensor shapes during model serving, device memory must be preallocated for both the LoRA ﻿AAA﻿ and LoRA ﻿BBB﻿ matrices for static multi-LoRA serving. In the illustration above, we assume that up to 8 LoRA adapters can be stored on device (max_loras = 8) and the batch size is 4. Users specify the LoRA adapter IDs in their requests. During inference, the adapter IDs are used to retrieve LoRA weights for LoRA computation as follows:
# Step 1: Retrieve LoRA weights from those preallocated in HBM
active_loras_A = loras_A[adapter_ids]
active_loras_B = loras_B[adapter_ids]

# Steps 2 and 3: Perform LoRA module computation
lora_A_output = torch.matmul(active_loras_A, input)
lora_output = torch.matmul(active_loras_B, lora_A_output)

# Step 4: Combine the output from the base module and LoRA
output = torch.add(lora_output, base_output)
The weights are gathered during each round of computation, including both context encoding and token generation.

Customers may require hundreds to thousands of adapters for their applications. Due to the limited number of LoRA adapters that can fit in HBM, we must utilize CPU memory to store additional adapters and dynamically swap these into HBM at runtime. This necessitates support for dynamic multi-LoRA serving.
Dynamic Multi-LoRA Serving (Beta Released to Databricks on 09/25/25)
Per customer requirements for dynamic multi-LoRA serving on TRN2, we developed the basic functionality to support dynamic swapping of adapters from CPU memory to HBM in NeuronxDistributed Inference (NxDI) in June 2025 as the second milestone. This is the first NxDI feature that enables dynamic data loading from CPU memory to HBM during model serving. 

With dynamic multi-LoRA serving, significant device memory does not need to be preallocated for LoRA adapters before execution begins, as adapters can be loaded on-demand and swapped in/out of device memory as needed. This enables serving hundreds or thousands of adapters by keeping only the “active” adapters in device memory while offloading or storing the remaining in a secondary memory such as CPU memory or disk. 
End-to-End Dataflow
In a typical setup to support hundreds of adapters, the majority of LoRA adapters are hosted in CPU memory and a subset of them are loaded into HBM. We present an example to highlight the flow of dynamically swapping an adapter from CPU memory to HBM. We only illustrate the LoRA matrix that is a linear layer and duplicated across all Neuron devices, where N = the tensor parallelism degree. 
image.png
In this example, LoRAs 1, 2, 3, and 4 are hosted in HBM, while all LoRAs (1-100) are hosted in CPU memory. Among the adapters in HBM, LoRA 1 has been least recently accessed. Suppose a user requests LoRA 4 and LoRA 5, but in this case, only LoRA 4 is in HBM. The weights of LoRA 5 therefore must be loaded from CPU memory to HBM dynamically at runtime. Following an adapter swap, LoRA 5 replaces LoRA 1.

Performance Bottlenecks in Multi-LoRA Serving
Our initial implementation of multi-LoRA was based on techniques for GPUs, i.e. the vLLM implementation of multi-LoRA serving. This caused the multi-LoRA serving performance on Neuron to be suboptimal. We analyzed the performance and identified the following opportunities for improvement:
Bottleneck #1: Performing the Same  LoRA Weight Transfer from CPU to Each Device
Suppose S is the LoRA adapter size, i.e. the total memory footprint of LoRA ﻿AAA﻿ and LoRA ﻿BBB﻿. Since the LoRA linear layer is duplicated across all N devices, while the other is partitioned, the LoRA adapter weight size on each Neuron device is S/2 + S/(2N) (TP degree = 32 → total memory of LoRA weights transferred = 16.5S). A memory transfer of duplicated data has to be performed to each Neuron device. The following table lists the memory transfer sizes for the Llama3.1 8B model. When the batch size = 4, more than 21GB data has to be transferred during an adapter swap.

Bottleneck #2: Duplicate LoRA Weight Gathers During Inference (Prefill and Decode)
Some HBM capacity is needed to host multiple LoRA adapters and gather selected LoRA adapters based on user requests for matmul . Since the gathered tensors are required for each LoRA computation, the tensor operation loras[adapter_ids] has to occur in both context encoding and the generation of each token. Because the token generation phase is memory-bound, gathering tensors incurs non-negligible latency overhead.

Bottleneck #3: Inefficient LoRA Adapter Swaps due to Small Memory Transfers
When a LoRA adapter needs to be swapped in from CPU memory to device, the LoRA weights need to be updated for each device, LoRA module, and layer. If there are 32 devices, 14 LoRA modules, and 32 layers, then a single adapter swap requires 32 x 14 x 32 = 14,336 memory transfers and each of these memory transfers are small and do not fully utilize the I/O bandwidth. Furthermore, the swaps to each device are performed sequentially. If the swaps to each device are performed in parallel, there are still 448 memory transfers that need to take place for each device. Fusing small tensors together into larger ones can better utilize the I/O bandwidth and reduce the number of memory transfers. Bottleneck #4: Blocking Memory Transfers between CPU and Device
In Neuron, memory transfers between CPU and device are performed via the following runtime function: 
nrt_tensor_write(const nrt_tensor_t *tensor, void *buf, size_t offset, size_t size)
This function is blocking, i.e. model execution taking place on the device must pause while the data transfer is taking place. This means every memory transfer incurs latency that adds to the end-to-end inference latency. These memory transfers can be overlapped with compute taking place on the device to reduce the end-to-end latency.

Optimizations for Multi-LoRA Serving on Neuron
Optimization #1: LoRA Linear Layer Sharding to Avoid Weight Duplication
In order to address the duplicated LoRA weights on multiple devices (and duplicate memory transfers), the first optimization is to shard the LoRA linear layers at the cost of extra communication to ensure all devices are up to date following weight updates. We pre-shard the LoRA weights, which are stored in CPU memory and dynamically loaded into HBM (some weights can be preallocated in device memory). Different chunks of data on CPU are transferred to different Neuron cores. On each Neuron core, the reserved HBM for LoRA adapters only stores the sharded chunk. Before LoRA computation, the LoRA weights are first selected from the reserved HBM, and gathered across devices via an allgather operation. 
Empty Image
# Step 1: Retrieve LoRA weights from those preallocated in HBM
active_loras_A = loras_A[adapter_ids]
active_loras_B = loras_B[adapter_ids]

# Step 2: allgather LoRA weights (assume LoRA A is sharded)
active_loras_A = allgather(active_loras_A)

# Step 3 and 4s: Perform LoRA module computation
lora_A_output = torch.matmul(active_loras_A, input)
lora_output = torch.matmul(active_loras_B, lora_A_output)

# Step 5: Combine the output from the base module and LoRA
output = torch.add(lora_output, base_output)
LoRA linear layer sharding can address the first performance bottleneck mentioned above. Since no LoRA weights are duplicated, the data volume of each LoRA adapter transferred from CPU to device is simply the LoRA adapter size S, rather than 16.5S with a TP degree of 32. In addition, this optimization also significantly increases the number of concurrent LoRA adapters that can fit in HBM.

It is important to note that a naïve implementation of LoRA linear layer sharding can cause the allgather operations to dominate latency due to the additional communication across devices.  We addressed this with the next optimization.
Optimization #2: Hierarchical HBM for LoRA Adapters
We observed that the LoRA weights are identical across rounds of computation, such as context encoding and the generation of each token. However, the current implementation performs an allgather of the LoRA weights in each round, causing dramatic latency regression. Based on this observation, we reserve part of HBM to store completed LoRA weights after performing the allgather in context encoding. These weights can be reused during token generation to avoid repeated allgather communications. 
Empty Image
This optimization preallocates two types of HBM for LoRA adapters: lora_memory that stores sharded LoRA weights and lora_memory_active that stores LoRA weights that have been gathered for the adapter IDs in a batch. This design leads to different operations for LoRA weights in different stages.
Dynamic Loading of LoRA Adapters
The LoRA weights on CPU (along with preallocated weights on the device) are sharded in advance, i.e. before execution begins. Each sharded chunk is transferred to lora_memory on different Neuron cores in the event that a LoRA adapter is requested and its data is not already allocated in HBM.
# N is the number of Neuron cores involved 
# modules is the list of all LoRA modules
for rank in range(N):
    for module in modules:
        lora_weight_cpu = lora_weights_cpu[rank][module]
        lora_memory[rank][module].copy_(lora_weight_cpu)
Context Encoding
After the LoRA weights are selected from lora_memory , they need an allgather operation to ensure each core has all of the LoRA weights (all updates performed on different devices) for computation. These full LoRA weights are stored in lora_memory_active .
# Step 1: Select LoRA weights
active_loras_A = lora_memory_A[adapter_ids]
active_loras_B = lora_memory_B[adapter_ids]

# Step 2: allgather LoRA weights (assume LoRA A is sharded)
lora_memory_active_A.copy_(allgather(active_loras_A))
lora_memory_active_B.copy_(active_loras_B)

# Step 3: Perform LoRA module computation
lora_A_output = torch.matmul(lora_memory_active_A, input)
lora_output = torch.matmul(lora_memory_active_B, lora_A_output)
Token Generation
The weights in lora_memory_active can be directly used for LoRA module computation without tensor selection or an allgather operation.
lora_A_output = torch.matmul(lora_memory_active_A, input)
lora_output = torch.matmul(lora_memory_active_B, lora_A_output)
This design avoids recurring tensor gather and allgather communications during token generation. Although they are still required in context encoding, they only incur low overhead because context encoding is compute-bound.

vLLM Integration (Neuron Plugin)
We have integrated dynamic multi-LoRA serving in NxDI with vLLM V1 for both offline and online inference and delivered a beta release of the artifacts to Databricks. We received feedback that they were able to replicate the functionality of multi-LoRA serving on Neuron. We also received feedback from the PMs that we should support streaming adapters (dynamically loading adapters e.g. from HuggingFace or disk) as well as running the base model only even when multi-LoRA serving is enabled. 
Optimization #3: LoRA Weight Tensor Fusion for Weight Transfers from CPU to Device
We observed that we can perform larger data transfers between the CPU host and Neuron device to better utilize the I/O throughput. On each device, the LoRA weight tensors are organized in the layout of [layer ID, LoRA module, adapter ID]. In order to transfer more data at a time, we can fuse all weight tensors corresponding to a given LoRA module for each adapter. More specifically, we are fusing across layers so that all individual layer weight tensors are collectively transferred together. For example, if there are 32 layers and 14 LoRA modules, we can perform 14 memory transfers (per device) that are 32x larger in size compared to the baseline which performs 448 memory transfers. With this optimization, we project a weight swap performance improvement of 20% for Llama3.1 8B, which translates into an E2E latency improvement of 1-2% over dynamic multi-LoRA serving without tensor fusion and a 10x memory throughput improvement.
fusion.png
Optimization #4: Overlap Computation with LoRA Weight Transfers
In NxDI, memory transfers can only be controlled through synchronous Neuron runtime tensor APIs and therefore it is challenging to overlap compute with memory transfers at the framework level. @Kelvin Ng is introducing ﻿SpikeAsync﻿, flexible Neuron runtime APIs to enable asynchronous operations. With this work, LoRA adapter swaps can be overlapped with prefill stages.
Future Directions
Leverage CPU Resources (e.g. Underutilized Host Memory) for Efficient Model Serving
Dynamic multi-LoRA serving is the first feature to perform memory transfers between CPU host and Neuron devices at runtime and consequently leverage CPU memory. Additionally, our work introduces optimizations to minimize the overheads of such transfers. This paves the way for future features on Neuron that rely on CPU offloading, which we present below.
KV Cache Offloading for Higher Prefix Cache Hit Rate
Offloading large KV caches to CPU memory can leave device memory for prefix caches and consequently enable higher prefix cache hit rates.
Empty Image
Weight Preloading of Multiple Models for Fast Model Switching
vLLM recently introduced ﻿Sleep Mode﻿ for fast model switching, i.e. models can be switched on the device without a full reload of weights. The first level of this mode leverages CPU offloading so that model weights are readily available in system memory and can be preloaded.
Empty Image
KV Cache Fusion for Data Transfers in Disaggregated Inference
The tensor fusion optimization described in Optimization #3: LoRA Weight Tensor Fusion for Weight Transfers from CPU to Device is not only applicable to multi-LoRA serving, but also disaggregated inference, where the KV cache data (slices of large tensors) must be transferred between the prefill and decode stages. In Neuron, the prefill worker must transfer data to the CPU host, which transfers data to the decode worker. Applying tensor fusion to the KV cache tensors will make the memory transfers between the CPU host and devices more efficient.
Empty Image
Adapter Prefetching
Based on access patterns, some adapter requests may become predictable and this knowledge can be used to swap in adapters from CPU memory ahead of time. Ideally, this prefetching of adapters takes place asynchronously (overlapped with compute) to reduce swap latency overhead.
Adapter Eviction Policy
With multiple adapters in use and limited device memory allocated for them, the eviction policy to determine which adapter on device to swap out becomes important for dynamic multi-LoRA performance. The current implementation has support for two eviction policies: Least Recently Used (LRU) and Least Frequently Used (LFU). Depending on the scenario, more intelligent eviction policies based on adapter access patterns may perform better.

How does LoRA Work?
Let ﻿W∈Rd×kW \in \mathcal{R^{d \times k}}W∈Rd×k﻿ represent the original model weights, which are pretrained and frozen during LoRA fine-tuning. Instead of learning a full update ﻿ΔW\Delta WΔW﻿, LoRA decomposes the weight matrix into two new, smaller matrices  ﻿A∈Rd×rA \in \mathcal{R^{d \times r}}A∈Rd×r﻿ and ﻿B∈Rr×kB \in \mathcal{R^{r \times k}}B∈Rr×k﻿, where ﻿ΔW=AB∈Rd×k\Delta W = AB \in \mathcal{R^{d \times k}}ΔW=AB∈Rd×k﻿ and rank ﻿r≪min(d,k)r \ll min(d,k)r≪min(d,k)﻿ (typically 8, 16, 32, or 64). Only ﻿AAA﻿ and ﻿BBB﻿ are updated during LoRA fine-tuning. For example, if the weight matrix has dimensions 4096x4096 and the rank is 16, instead of training ~16M parameters, LoRA only trains ~131K parameters (two matrices: 4096x16 + 16x4096).

For multi-LoRA serving, the ﻿ABABAB﻿ pairs are swapped for different requests. During inference, the weight matrix ﻿WWW﻿ and ﻿ABABAB﻿ can be computed separately then added together. Alternatively, adapters can be merged by computing ﻿W+ABW + ABW+AB﻿ offline and using the merged weights (to reduce runtime overhead). 

LoRA is based on the hypothesis that model adaptations have low "intrinsic dimensionality," i.e. the task-specific changes needed lie in a much smaller subspace than the full parameter space. Research has demonstrated that most adaptation information can be captured with ranks as low as 1-8 for many tasks.
